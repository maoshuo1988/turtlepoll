# 预测市场标签查询（PG-only）- 高性能/高可用设计方案

前提约束（已确认）：本项目**只使用 PostgreSQL**，不引入 Redis、Prometheus 等外部依赖。

目标：提供一个稳定、高并发、低延迟的“查询预测市场所有标签（tags）”接口；同时确保与现有“预测市场同步写入 `t_predict_context`（`PredictContext.Tags`）”链路**业务闭环**，修改方案不遗漏。

本文档按设计要点分节：需求、接口契约、表结构与索引、从 `t_predict_context` 聚合/物化到 tag 表的策略、刷新/一致性、可用性与降级、运维与演进。

## 1. 需求与约束

- 功能：返回“预测市场标签”列表，每个标签包含 slug/name/marketCount/updatedAt 等字段；支持按 slug 批量过滤、按 name 搜索、分页与排序。
- 性能：高并发读（目标 QPS 1k+，突发可承受 5k），P95 < 120ms（PG-only 场景下主要靠索引/聚合表与读副本）。
- 可用性：99.9% 可用；PG 异常时可降级（更多依赖读副本/限流/关闭昂贵功能），不承诺“离线缓存数据”。
- 一致性：允许最终一致性（标签来自同步的 `t_predict_context.tags`，天然是异步更新）。
- 闭环要求：Polymarket 同步已经在写 `PredictContext.Tags`（逗号分隔），标签接口必须以此为数据源或可追溯来源。

## 2. 接口契约（API 设计）

- 路径：GET /api/predict/tags
- 权限：公开接口（可选限流/身份识别）
- 查询参数：
  - page (int, default 1)
  - pageSize (int, default 20, max 200)
  - q (string, 可选，name 模糊搜索)
  - slugs (string, 可选，逗号分隔的 slug 列表，用于批量精确过滤)
  - sort (string, 可选："updatedAt"/"marketCount"，默认 updatedAt desc)
  - includeCounts (bool, 可选，是否返回 marketCount；PG-only 下建议默认 true 但依赖 tag_stat 表，避免在线聚合)

- 返回体（成功，HTTP 200）：

```json
{
  "success": true,
  "data": {
    "list": [
      {"id": 1, "slug": "crypto", "name": "加密货币", "marketCount": 123, "updatedAt": 1680000000},
      ...
    ],
    "count": 1234,
    "page": 1,
    "pageSize": 20
  }
}
```

错误：统一返回 {success:false,message:...}，并记录 code 以便追踪。

契约说明（PG-only）：

- `marketCount` 不建议在查询时从 `t_predict_context`/market 表在线 `COUNT(DISTINCT ...)` 计算；应通过**物化/聚合表**输出。
- 若 `includeCounts=0`，可只查 tag 维表以进一步降级。

## 3. 现状核对：同步写入 `t_predict_context`（闭环基线）

已核对代码（关键点）：

- 模型：`internal/models/models.go`
  - `PredictContext.MarketId` 为 **uniqueIndex**（一对一）
  - `PredictContext.Tags` 为 `varchar(1024)`，内容为**逗号分隔**标签 slug（小写）
- 同步：`internal/services/polymarket_sync_service.go`
  - `upsertMarketAndContext()` 在 upsert `PredictMarket` 后，会写/更新 `PredictContext.EventName` 与 `PredictContext.Tags`
  - tags 由 `baseTags + market.Tags[].Slug` 合并去重、转小写后 `strings.Join(tags, ",")`

因此：标签体系的“事实来源”= `t_predict_context.tags`（来自同步）。本方案在此基础上做“聚合/物化”，避免每次请求做字符串拆分与全表扫描。

## 4. 数据模型（PG-only）——从 `t_predict_context` 物化到 tag 表

### 4.1 tag 维表（物化产物）

建议新增（或按你们命名规范落到 `t_predict_tag`）：

```
t_predict_tag
 - id (pk)
 - slug (unique)
 - name (可选，若暂无中文名可先=slug)
 - last_seen_at (int64)    -- 最近一次在任意 PredictContext.Tags 出现的时间
 - updated_at (int64)
```

索引建议：

- unique：`(slug)`
- `idx_predict_tag_updated_at (updated_at desc)`
- 若需要搜索 `q`：
  - 简单 like：`idx_predict_tag_slug (slug)` + `idx_predict_tag_name (name)`
  - 高性能模糊：启用 `pg_trgm`，`gin(name gin_trgm_ops)`（可选扩展，但仍属于 PG）

### 4.2 tag 统计表（强烈推荐，避免在线聚合）

```
t_predict_tag_stat
 - tag_id (pk / unique)
 - market_count (int)
 - refreshed_at (int64)
```

说明：这里的 market_count = 该 tag 在多少个 PredictContext 中出现（等价于 marketId 去重计数）。

索引建议：

- `pk(tag_id)`
- `idx_predict_tag_stat_market_count (market_count desc)`（用于排序）

## 5. 物化/刷新策略（从 `t_predict_context.tags` 聚合）

关键约束：`PredictContext.Tags` 是逗号分隔字符串；PG 侧聚合需解析。

### 5.1 刷新方式选型

推荐“两段式”，兼顾简单与性能：

1) **周期性全量刷新（可控频率）**
   - 每 N 分钟（例如 10~30 分钟）在后台任务里执行：
     - 扫描 `t_predict_context.tags`，用 `string_to_array(tags, ',')` 拆分 + `unnest` 拉平
     - upsert 到 `t_predict_tag`
     - 计算 `market_count` 写入 `t_predict_tag_stat`
   - 优点：实现简单、最终一致性明确；缺点：刷新窗口内数据稍旧

2) **增量刷新（可选增强）**
   - 记录 `t_predict_context.update_time` 的水位 `last_context_update_time`
   - 每次只处理 update_time > 水位的 context 行，解析 tags 增量 upsert
   - 同时增量更新 stat（可采用临时表聚合再 merge）

在不引入消息队列的前提下，优先落地 1)，后续量大再做 2)。

### 5.2 刷新 SQL（概念示意）

> 这里只写思路不落代码（评审阶段）。

- tag 维表来源：
  - `SELECT DISTINCT lower(trim(tag)) AS slug FROM (SELECT unnest(string_to_array(tags, ',')) tag FROM t_predict_context WHERE tags<>'' ) s WHERE slug<>''`
- 统计来源：
  - `SELECT slug, COUNT(DISTINCT market_id) FROM (context join unnest tags) GROUP BY slug`

### 5.3 并发与锁

- 刷新任务使用事务 + 分阶段写入，避免长事务锁住核心表：
  - 可使用临时表/中间表（`t_predict_tag_stat_tmp`）先算好，再 `INSERT ... ON CONFLICT DO UPDATE` 合并
- 避免对 `t_predict_context` 做 `FOR UPDATE`；只读扫描 + 依赖 MVCC

## 6. 标签查询接口的查询路径（性能关键）

请求 `GET /api/predict/tags`：

- 常规 list：读取 `t_predict_tag`（小表）
- includeCounts=1：left join `t_predict_tag_stat`
- sort=marketCount：order by stat.market_count desc nulls last
- slugs=...：where slug in (...)
- q=...：where name ilike %q% 或 trigram

核心原则：**线上请求不再碰 `t_predict_context` 做字符串拆分与聚合**，避免全表扫描。

## 7. 高可用设计（PG-only）

### 7.1 读写分离

- 同步写入 `t_predict_market/t_predict_context` 与刷新写入 `t_predict_tag*`：走主库
- `GET /api/predict/tags`：优先走只读副本（若配置了 replica）

### 7.2 降级策略

当 DB 压力过大或 replica 延迟过高：

- 强制 `includeCounts=0`
- 将 `pageSize` 上限从 200 降低到 50
- 禁用 `q` 模糊搜索（仅允许前缀或 slug 精确）

## 8. 监控/告警（PG-only，可选落地）

不引入 Prometheus 的前提下，推荐：

- 应用日志：记录接口耗时/慢查询（>200ms）与参数
- PG 侧：开启 `pg_stat_statements`（若可用）观察 top queries
- 关键指标：刷新任务耗时、刷新失败次数、最后刷新时间（写到 `t_predict_tag_stat.refreshed_at` 可观测）

## 9. 交付与验收（闭环检查点）

### 9.1 闭环必须验证

1) 预测市场同步（Polymarket）仍会写 `PredictContext.Tags`（已在 `upsertMarketAndContext` 实现）
2) 刷新任务能从 `t_predict_context.tags` 产出 `t_predict_tag` 与 `t_predict_tag_stat`
3) 标签接口只读 `t_predict_tag*`，并能覆盖：分页/过滤/排序/includeCounts

### 9.2 数据一致性验收

- 手工抽样：任选一个 `PredictContext` 的 tags，确保在 `t_predict_tag` 中存在
- 统计抽样：对比 `context + unnest` 的某个 tag 的 count 与 `t_predict_tag_stat.market_count` 一致


## 10. 分页、搜索与排序实现细节（PG-only补充）

name 搜索（`q`）：

- 简单：`ILIKE '%q%'`（配合 trigram 可显著提升）
- 若不启用 trigram：可降级为 `slug 前缀匹配` 或限制 q 最小长度（例如 >= 2）

排序：

- updatedAt：直接对 `t_predict_tag.updated_at` 排序
- marketCount：对 `t_predict_tag_stat.market_count` 排序（避免在线 count）

## 11. 安全与合规

- 公共标签数据为公开信息，无需敏感数据处理
- 运营写接口需要鉴权与审核日志，避免恶意创建/篡改标签
