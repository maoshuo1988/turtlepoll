# 预测市场标签查询 - 高性能/高可用设计方案

目标：提供一个面向产品与后端服务的稳定、高并发、低延迟的“查询预测市场所有标签（tags）”接口，支持分页与过滤，能承受突发流量并能在 DB 异常时优雅降级。

本文档按设计要点分节：需求、接口契约、数据模型、数据流与缓存策略、一致性权衡、扩展与分片、容灾/降级/回退、监控/告警与运维、演进建议。

## 1. 需求与约束

- 功能：返回“预测市场标签”列表，每个标签包含 id/slug/name/marketCount/updatedAt 等字段，支持按 slug 批量过滤、按 name 模糊搜索、分页与排序。
- 性能：支持高并发读（目标 QPS 1k+，突发可承受 5k），P95 响应 < 120ms（含缓存命中路径）。
- 可用性：99.9% 可用，DB 异常时能提供最后一致性的缓存数据（短期内），并能快速回退与告警。
- 一致性：允许短时间的最终一致性（后台同步 / cache TTL），写操作（标签变更）不频繁。
- 可扩展性：支持横向扩容、读写分离、地域分布（必要时）。

## 2. 接口契约（API 设计）

- 路径：GET /api/predict/tags
- 权限：公开接口（可选限流/身份识别）
- 查询参数：
  - page (int, default 1)
  - pageSize (int, default 20, max 200)
  - q (string, 可选，name 模糊搜索)
  - slugs (string, 可选，逗号分隔的 slug 列表，用于批量精确过滤)
  - sort (string, 可选："updatedAt"/"marketCount"，默认 updatedAt desc)
  - includeCounts (bool, 可选，是否返回 marketCount，计算稍昂贵）

- 返回体（成功，HTTP 200）：

```json
{
  "success": true,
  "data": {
    "list": [
      {"id": 1, "slug": "crypto", "name": "加密货币", "marketCount": 123, "updatedAt": 1680000000},
      ...
    ],
    "count": 1234,
    "page": 1,
    "pageSize": 20
  }
}
```

错误：统一返回 {success:false,message:...}，并记录 code 以便追踪。

契约说明：为减少 DB 负担，建议 `includeCounts` 默认 false；若前端需要列表显示 marketCount 建议走单独异步接口或由缓存定期预计算。

## 3. 数据模型（后端）

- 数据库表（示意）：

```
predict_tags
 - id (pk)
 - slug (unique)
 - name
 - meta (jsonb) -- 额外字段
 - market_count (int) -- 可选，若不实时计算则由同步任务更新
 - updated_at (timestamp)
```

设计点：将计算量大的字段（market_count）由后台同步/批处理更新并写入列（避免查询时 join 或聚合），以利缓存与索引。

## 4. 数据流与缓存策略

总体思路：读主缓存（Redis），二级回源 DB；写操作（标签新增/更新/删除）走写库并触发缓存 invalidation & async update。

- Redis 热点缓存（一级）：
  - Key 设计：
    - `predict:tags:list:{page}:{pageSize}:{q}:{sort}:{includeCounts}` -- 用于常规分页列表缓存
    - `predict:tags:slug:{slug}` -- 单个 tag 缓存
  - TTL：短至 30s-300s（根据流量/一致性要求可调）
  - 缓存值：JSON 序列化的列表/对象、并缓存 metadata（count/timestamp）

- 本地内存 LRU（二级，可选）：在 API 层做一次本地缓存以减少 Redis 负载，适配单机高 QPS 场景

- 回源 DB 策略：
  - 缓存未命中 -> 查询读库（Read Replica）以减轻主库压力
  - 读库延迟允许最终一致性：若数据非常新，可从主库强制读取

- 写入与失效：
  - 标签变更（create/update/delete）经过事务写主库后：
    - 同步更新 `predict:tags:slug:{slug}`
    - 异步发布消息（Kafka/RabbitMQ）到 `tag-updated` topic，由消费者更新分页缓存与 market_count（若需要）或清除对应 list cache
  - 当收到消息，消费者按策略更新/删除 Redis 中相应 list keys（或使用版本号/namespace 方案避免 N keys invalidation）

### 版本号/namespace 技术

为避免对所有 list keys 逐一删除，可使用 `namespace` 或 `version` 方案：

- 在 Redis 中维护 `predict:tags:version`（整数）
- list key 为 `predict:tags:list:{version}:{page}:...`，标签更新时只需要 `INCR predict:tags:version`，下次请求会读取新 version 的 key，从而自动失效旧缓存（但旧缓存仍存在直到 TTL 过期）

优点：极快的失效（O(1)）；缺点：对应的旧 keys 不会立即回收（可在后台定期清理）。

## 5. 分页、搜索与排序实现细节

- 精确按 slug 批量过滤：优先通过 Redis 的 `predict:tags:slug:{slug}` 批量读取，若部分命中再回源 DB 批量查询以补全。
- name 模糊搜索（`q`）：
  - 若需高性能：同步将标签建立到搜索引擎（Elasticsearch / Meilisearch / Typesense），`q` 走搜索引擎并返回 ids，再批量从缓存/DB 拉取完整对象。
  - 简单需求：Postgres `ILIKE` + 合理索引（适合标签量小于百万）
- 排序：尽量在缓存层保存已排序好的数据（按 updatedAt/marketCount 预排序），避免每次回源排序

## 6. 可用性与容灾

- 多副本部署：服务水平扩展，Redis 主从或 cluster，DB 主从分离，查询走只读 Replica
- 灾备策略：跨可用区或跨区域部署读副本；关键服务（cache 消费/消息）要支持重试与幂等
- 降级策略：当 Redis/DB 不可用
  - Redis 出问题但 DB 可用：直接回源 DB（读库），但要限流（保护性阈值），并上报告警
  - DB 全不可用：尝试读取最后有效缓存（存在本地内存或 CDN 缓存）；若没有，返回可降级响应：{"success":true,"data":{"list":[],"count":0},"message":"部分数据降级，仅展示缓存结果"} 或 HTTP 503

## 7. 一致性与延迟权衡

- 标签变更频率低，允许最终一致性：写操作同步更新主库并发布事件，消费端异步更新 cache 与 market_count
- 对于强一致性场景（管理面板需要立即可见新标签），管理接口可在写后直接从主库读取并返回最新数据（绕过缓存）

## 8. 监控、指标与告警

- 指标（Prometheus）：
  - API QPS / latency (P50/P95/P99)
  - Redis hit ratio for tag list / item
  - DB replica lag
  - Cache invalidation events/sec
  - Message queue publish/consume lag

- 告警规则：
  - Redis hit ratio < 60% for 5m
  - DB replica lag > 5s
  - API error rate > 1% for 1m
  - Consumer lag > 10s

## 9. 灰度、回滚与演进策略

- 首次上线：Feature flag + 流量渐进（10% -> 50% -> 100%）
- 回滚：在 redis namespace 方案下，回滚只需恢复 `predict:tags:version`（或回到上一个稳定版本），并在后台回填数据
- 逐步演进：
  - 初期：实现 Redis list cache + read-replica 回源
  - 中期：引入 search engine 支持 `q` 模糊搜索
  - 长期：支持多区域部署与 CDN 缓存边缘化（tags 列表可放 CDN）

## 10. 安全与合规

- 公共标签数据为公开信息，无需敏感数据处理
- 运营写接口需要鉴权与审核日志，避免恶意创建/篡改标签
