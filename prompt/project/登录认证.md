
# 登录认证与人机校验（纯本地）：滑块拼图验证码 + 风控触发式

> 目标：替换当前“旋转图片验证码（angle/rotate）”，把登录/注册/找回密码等流程的人机校验变成 **更直观的滑块拼图**，并且 **默认不打扰**（只有触发风控才要求验证码）。
>
> 边界：本方案不依赖外部服务（无 Turnstile/hCaptcha），不引入 Redis 等中间件；风控数据落 PostgreSQL（允许少量进程内缓存作为加速/降载）。

---

## 1. 现状与约束（为了兼容既有业务表单）

项目当前在登录/注册/找回密码等接口里，提交以下字段（见 `docs/api/user.md`）：

- `captchaId`: string
- `captchaCode`: string
- `captchaProtocol`: int

服务端在反垃圾策略统一校验（见 `internal/spam/captcha_strategy.go`）：

- `captchaProtocol=2`：旋转验证码 `internal/pkg/captcha.Verify(captchaId, captchaCode)`（`captchaCode` 实际是角度）
- 其他：字符验证码 `github.com/dchest/captcha`

**兼容原则**：

1) 新滑块拼图验证码仍复用 `captchaId/captchaCode/captchaProtocol` 三件套，避免改动所有用户相关 API；
2) 新方案要支持“按需触发”（风控触发式），不能再要求每次都做交互；
3) 保留旧 rotate 作为灰度兜底。

---

## 2. 方案总览（你要什么 + 我们怎么落地）

### 2.1 交互层：滑块拼图验证码（比旋转更直观）

**用户看到的交互**：

1. 默认正常登录不出现验证码；
2. 只有当服务端判断“需要人机校验”时，前端弹出滑块拼图；
3. 用户拖动滑块，让缺口与拼图对齐；
4. 校验通过后，继续提交登录/注册表单。

**我们要实现的服务端能力**：

- 生成 puzzle challenge：返回背景图、拼图块图、滑块轨道信息、challengeId（即 `captchaId`）
- 校验 puzzle answer：校验 `captchaCode`（本质是“你拖到哪儿 + 行为轨迹 + 时间”），并做一次性消费/过期控制

### 2.2 策略层：风控触发式（默认不打扰）

**核心变化**：验证码从“固定步骤”升级为“风险控制手段”。

- 低风险：直接放行（不强制验证码）
- 中风险：要求滑块拼图
- 高风险：要求滑块拼图 + 强限流/临时冻结（可选）

## 3. 接口契约（建议新增/改造点）

> 说明：你当前已有 `GET /api/captcha/request` 和 `GET /api/captcha/request_angle`。
> 为了最小侵入，建议新增「滑块拼图」专用的 `request_puzzle`，并在业务表单校验时增加 `captchaProtocol=4` 的分支。

### 3.1 获取滑块拼图 challenge

**接口**：`GET /api/captcha/request_puzzle`

**响应（示例字段，按你项目风格落 JSON）**：

- `captchaId`: string（challengeId，后续提交到业务表单的 `captchaId`）
- `captchaProtocol`: 4（固定）
- `backgroundImage`: string（base64 或 URL，建议 base64 以免静态文件泄露）
- `pieceImage`: string（base64 或 URL）
- `backgroundWidth`: int
- `backgroundHeight`: int
- `pieceWidth`: int
- `pieceHeight`: int
- `pieceY`: int（拼图块的 y 固定，减少难度）
- `expireIn`: int（秒）
- `nonce`: string（可选；用于防重放/签名）

**实现要点**：

- challenge 记录必须含：正确 x（服务端保存/加密存储）、过期时间、是否已使用、请求方 ip/ua hash（可选）
- 图片生成要有随机噪声/干扰纹理，避免模板匹配

### 3.2 校验方式：仍然走“业务表单统一校验”

你现有流程是：登录/注册表单携带 `captchaId/captchaCode/captchaProtocol`，反垃圾策略里统一校验。

本方案建议新增：

- `captchaProtocol=4`：滑块拼图

表单字段复用方式：

- `captchaId`: 来自 `/api/captcha/request_puzzle` 返回的 challengeId
- `captchaProtocol`: 固定 4
- `captchaCode`: 建议传 JSON 字符串（兼容 string 类型），比如：
	- `{"x":123,"t":1680,"track":[[0,0,0],[3,0,16],...]}`
	- 或更紧凑的编码：`x=123&t=1680&track=...`

**服务端校验要点**：

- 允许误差：例如 `abs(x - expectedX) <= 3~5 px`
- 校验轨迹合理性（简单规则即可）：
	- 总耗时必须在合理区间（比如 300ms ~ 10s）
	- 轨迹点数量、速度变化不要“过于机器”（避免直线匀速）
- challenge 必须一次性消费（verify 成功/失败都可标记 used，失败可允许同 challenge 重试 1 次）
- 校验失败应计入风控计数

## 4. 风控触发式：规则、阈值与数据库落地（PG-only）

### 4.1 我们要解决的两个问题

1) **什么时候要求验证码？** —— 不能打扰正常用户，但要挡住爆破、撞库、批量注册。
2) **什么时候强限流/冻结？** —— 防止攻击者反复请求 challenge 或反复提交验证码答案。

### 4.2 建议的最小可用风控信号

按“统一入口 + 规则简单 + 可解释”为原则，先做这些：

- 登录失败（密码错误/账号不存在/验证码失败）
- 注册失败（手机号/邮箱校验失败、验证码失败）
- 验证码 challenge 请求频率过高

维度建议（key）：

- `ip`（必要）
- `username/email/phone`（登录/注册场景可用）
- `uaHash`（可选，作为辅助）

### 4.3 触发策略（可直接用的默认阈值）

> 下面阈值是“保守、易落地”的起点，后续可配。

- **触发验证码（needCaptcha=true）**
	- 同 IP 在 10 分钟内登录失败次数 >= 5
	- 或同 username/email 在 10 分钟内登录失败次数 >= 3
- **提升风险（needCaptcha=true + 加强限流）**
	- 同 IP 在 10 分钟内 login/register 请求 >= 60
	- 或同 IP 在 10 分钟内验证码 challenge 请求 >= 20
- **临时冻结（可选）**
	- 同 IP 在 10 分钟内验证码校验失败 >= 10 → 冻结 10 分钟（只允许查看，不允许登录/注册/发码）

### 4.4 数据表建议（便于查询/衰减/审计）

建议新增一张“滑动时间窗口计数”表，避免每次都扫日志。

#### 表：`t_risk_event_counter`（建议）

- `id` bigint pk
- `key_type` varchar（`ip`/`username`/`email`/`phone`）
- `key_value` varchar
- `scene` varchar（`login`/`register`/`captcha`）
- `event` varchar（`fail`/`success`/`captcha_required`/`captcha_fail`/`captcha_pass`/`request`）
- `window_start` timestamptz（按分钟或 10 分钟 round）
- `count` int
- `created_at`/`updated_at`

索引建议：

- `(key_type, key_value, scene, event, window_start)` unique（用于 upsert 计数）
- `(window_start)`（便于清理过期窗口）

#### 清理任务

- cron 每小时清理：`window_start < now() - interval '7 days'`

> 备注：如果你倾向更轻量，也可以用 `t_risk_state` 存“当前累计 + 过期时间”，但可解释性和审计性会弱一些。

### 4.5 风控判定的返回方式（建议）

业务接口在判定“需要验证码”时，建议返回：

- HTTP 200 + `code != 0` 的业务错误码（沿用你项目风格）
- 错误码：例如 `CaptchaRequired`
- 附加信息：`needCaptcha: true` + `captchaProtocolList: [4]`

前端收到后：

1) 先请求 `/api/captcha/request_puzzle`
2) 让用户完成拼图
3) 将 `captchaProtocol=4` + `captchaId/captchaCode` 回填原表单重试

## 5. 前端实现要点（避免“做了但不好用”）

### 5.1 组件行为

- 默认不渲染验证码
- 接到 `needCaptcha=true` 时再弹出（modal / bottom sheet）
- 拼图成功后自动关闭，并把验证码结果写入表单

### 5.2 交互细节

- 失败提示要明确："没对齐"/"操作太快"/"已过期，请重试"
- 过期自动刷新 challenge
- 移动端拖动要防止页面滚动干扰（touch-action）

### 5.3 重试与刷新

- 允许用户在同一次弹窗里刷新 challenge（换一张图）
- 验证失败 1~2 次后强制刷新 challenge（防止暴力枚举 expectedX）

---

## 6. 安全与对抗建议（不追求完美，但要“够用”）

### 6.1 Challenge 存储

- 服务端不要把 expectedX 明文下发
- challengeId 要难猜（uuid/随机串）
- challenge 必须绑定过期时间 + 一次性消费

### 6.2 防重放

- `captchaCode` 建议包含 `nonce` 或 `t`（时间戳）
- 服务端校验 “提交时间 - challenge 创建时间” 合理

### 6.3 防暴力枚举

- 校验失败次数达到阈值：强制换 challenge
- 对 `/api/captcha/request_puzzle` 做限流（按 ip）

---

## 7. 迁移与灰度（兼容旧 rotate，逐步切换）

### 7.1 协议号规划

- `captchaProtocol=2`：旧旋转（保留兜底）
- `captchaProtocol=4`：新滑块拼图（本方案）

### 7.2 上线步骤（建议顺序）

1) 后端先支持 `captchaProtocol=4` 校验 + 提供 `/api/captcha/request_puzzle`
2) 前端加“风控触发式弹窗”逻辑，但先保持不开启（feature flag）
3) 灰度开启：只对“触发风控”的用户启用滑块拼图；其他用户仍无验证码
4) 观察风控命中率、登录转化、失败率
5) 逐步下线 rotate（或保留仅做兜底）
