## 1. 目标与范围

### 目标

1. 管理员可以给用户账户铸币（充值/增发）。
2. 用户使用金币对预测市场下注。
3. 每个预测市场为二元选项：A / B。
4. 系统实时维护两个方向的下注池，并据此计算赔率。
5. 为避免早期赔率极端，系统提供“虚拟底池（baseA/baseB）”稳定赔率。
6. 下注时锁赔率；结算时不根据最新赔率重算。
7. 形成可审计的资金流水（扣款、派奖、退款）。

### 一期不做（但预留）

- 撤单/取消下注（`PredictBet.Status=CANCELED` 仅预留）。
- 多选项市场（>2 个 outcome）、阶梯赔率、盘口深度。
- 平台手续费/抽成。

## 2. 核心概念与数据模型（业务口径）

### 2.1 预测市场：`PredictMarket`

一条市场记录代表一个可下注的事件（当前项目里通常与赛程或外部市场绑定）。

关键字段（对应 `internal/models/models.go`）：

- `id`
- `title`：展示标题
- `status`：`OPEN / CLOSED / SETTLED`
- `closeTime`：下注截止时间（秒级时间戳；0 表示不限制）
- `result`：预留字段（未来用于落地最终结果/或映射）
- `externalKey`：外部 key（对接外部市场时用）

二元池字段（本模块核心）：

- `baseA/baseB`：系统虚拟底池（默认 500/500）
- `poolA/poolB`：用户真实下注累计池

外部市场结算字段（对接 Polymarket Gamma 时）：

- `resolved`、`resolvedOutcomeId`、`resolvedOutcomeName`、`resolvedAt`

> 注意：当前表模型同时支持“内部二元下注池”与“外部市场结算信息”。
> 后续需要明确：外部 outcome（YES/NO 等）如何映射到本地 A/B（见本文第 8 节待明确项）。

### 2.2 市场上下文：`PredictContext`

与 `PredictMarket` 一对一，承载 UI 展示数据：

- `eventName`、`imageUrl`、`detail`
- `heat`：热度
- `tags`：逗号分隔字符串，如 `"wc,final"`

### 2.3 下注单：`PredictBet`

下注单代表一次下注行为；一个用户可以对同一市场下多笔单。

关键字段：

- `userId`、`marketId`
- `option`：`A` / `B`
- `amount`：下注金币
- `odds`：**下单时锁定赔率**（范围 1.2~5.0）
- `effA/effB`：下单时有效池（用于审计/展示）
- `status`：`OPEN / SETTLED / CANCELED`
- `settleResult`：`WIN / LOSE / DRAW(预留)`
- `payout`：派奖金币（通常 `floor(amount * odds)`）

### 2.4 金币账户与流水：`UserCoin / UserCoinLog`

- `UserCoin`：余额（可预留冻结余额）
- `UserCoinLog`：流水（bizType 例如 `MINT/BET/SETTLE/REFUND`）

## 3. 业务流程（端到端闭环）

### 3.1 市场创建/来源

市场可来自：

1. 站内创建（未来可加 admin 接口）。
2. 赛事赛程同步（MatchSchedule → PredictMarket）。
3. 外部市场同步（Polymarket Gamma → PredictMarket + PredictContext）。

要求：

- 同一个外部市场需幂等 upsert（避免重复）。
- 同步写入 `PredictContext.tags`，作为标签/聚合的事实来源。

### 3.2 用户下注（PlaceBet）

入口（已实现）：`POST /api/coin/bet`（form：`marketId`、`option(A/B)`、`amount`）

规则（与当前实现保持一致）：

1. 校验登录、参数合法。
2. 校验 `PredictMarket.status == OPEN`。
3. 校验 `closeTime`：若 `closeTime>0` 且 `nowSec>=closeTime` 则拒绝下注。
4. 基于当前池计算赔率（见第 4 节），并锁定到 `PredictBet.odds`。
5. 事务内创建 `PredictBet`。
6. 扣减金币并记录流水（`bizType=BET`，amount<0）。
7. 更新市场池 `poolA/poolB`。

一致性与并发：

- 下单、扣款、更新池必须在同一事务。
- 对 `PredictMarket` 加行锁（`FOR UPDATE`）避免并发下注丢失更新。

### 3.3 结算（Settle）

入口（已实现）：`POST /api/coin/settle`（form：`marketId`）

结算原则：

- 必须幂等：同一笔 `PredictBet` 只能派发一次。
- 不重算赔率：使用下注单 `bet.odds`。
- 未具备最终结果的市场不可结算。

结果来源（需要统一口径）：

1. 站内管理员裁决（未来）。
2. 赛程结果同步（例如 FINISHED）。
3. 外部市场同步（Polymarket resolvedOutcome）。

### 3.4 退款/作废（预留）

当市场被 void/取消：

- 未结算下注原路退回（`bizType=REFUND`）。
- 对 bet 做状态标记（可选：SETTLED 或单独 REFUNDED/CANCELED）。

## 4. 赔率模型（虚拟底池 + clamp）

### 4.1 设计动机

- 早期真实下注池很小，赔率波动极端。
- 用 `baseA/baseB` 做虚拟底池稳定赔率。
- 为避免前端显示 1.01 或 100 这种极端值，对赔率做区间限制：`1.2 ~ 5.0`。

### 4.2 计算公式

定义：

- `baseA/baseB`：系统虚拟底池（默认 500/500）
- `poolA/poolB`：用户真实下注累计
- `effA = baseA + poolA`
- `effB = baseB + poolB`
- `total = effA + effB`

原始赔率：

- `oddsA_raw = total / effA`
- `oddsB_raw = total / effB`

锁定赔率（展示赔率）：

- `oddsA = clamp(oddsA_raw, 1.2, 5.0)`
- `oddsB = clamp(oddsB_raw, 1.2, 5.0)`

### 4.3 示例：早期有虚拟底池时的表现

假设：

```text
baseA = 500, baseB = 500
poolA = 0,   poolB = 0
```

初始：

```text
effA = 500
effB = 500
total = 1000

oddsA_raw = 2.0
oddsB_raw = 2.0
```

第一个用户押 A 100：

```text
poolA = 100, poolB = 0
effA = 600,  effB = 500, total = 1100

oddsA_raw ≈ 1100/600 ≈ 1.83
oddsB_raw = 1100/500 = 2.20
```

再有人押 B 400：

```text
poolA = 100, poolB = 400
effA = 600,  effB = 900, total = 1500

oddsA_raw = 1500/600 = 2.50
oddsB_raw ≈ 1500/900 ≈ 1.67
```

### 4.4 关键规则：下注时锁赔率

- 下单时写入 `PredictBet.odds`。
- 结算时用锁定赔率计算 payout，不按最新赔率重算。

## 6. 状态机与约束

### 6.1 PredictMarket 状态

- `OPEN`：可下注
- `CLOSED`：不可下注（但未结算）
- `SETTLED`：已结算（具备最终结果）

### 6.2 PredictBet 状态

- `OPEN`：未结算
- `SETTLED`：已结算
- `CANCELED`：预留

### 6.3 时间戳约定

- `closeTime/createTime/updateTime/settleTime` 统一使用**秒级时间戳**。
- 外部输入若为毫秒，入库前需转换。

---

## 7. 数据一致性与对账

### 7.1 可审计

每一笔资金变化必须可追溯：

- 下注：扣款 + `UserCoinLog(bizType=BET)`
- 结算：派奖 + `UserCoinLog(bizType=SETTLE)`
- 铸币：入账 + `UserCoinLog(bizType=MINT)`
- 退款：返还 + `UserCoinLog(bizType=REFUND)`

### 7.2 幂等

- 同一笔 bet 结算不得重复派发。
- 同一市场的重复结算调用应安全（重复调用不产生重复资金变化）。

## 8. 风险点与待明确项（设计 → 开发必须对齐）

1. **结果来源到 A/B 映射**：
   - 外部 `resolvedOutcomeName/Id` 如何映射到本地 `A/B`？
   - 若市场来自足球 1x2，是否拆成多个二元市场，还是保留 1x2 玩法？
2. **CLOSED vs SETTLED 定义**：
   - 是否需要定时任务按 `closeTime` 自动 OPEN→CLOSED？
   - SETTLED 的准入条件是什么（外部结果到达/管理员裁决）？
3. **风控与限额**：
   - 最小下注额、单笔/单日限额、余额不足提示文案等。
4. **clamp 的产品后果**：
   - clamp 会改变“真实池比例”对应的期望收益，这是产品策略，需要明确接受。
5. **void/退款口径**：
   - void 退本金；`baseA/baseB` 仅用于计算，不参与资金流。
