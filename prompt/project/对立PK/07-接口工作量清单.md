# 对立PK：接口工作量清单

## 范围说明

本文只梳理接口层工作量，包含用户端接口、管理端接口、现有接口改动、定时任务，以及每个接口背后的主要服务、校验、数据读写与返回内容。

工作量等级：S=简单查询或薄封装；M=有业务校验和多表读写；L=涉及事务、幂等、结算、状态推进或复杂聚合。

## 用户端新增接口

### 1. PK话题列表

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/pk/topics` |
| 工作量 | M |
| 作用 | 获取所有启用的对立PK话题，供列表页展示 |

要做的事：

- 查询 `PKTopic` 启用列表，按 `sort/id` 排序。
- 批量加载每个话题的当前 `PKRound`、当前 `PKSeason`。
- 返回当前阶段、倒计时、A/B 热度、A/B 下注人数、赔率。
- 返回上局胜方、守擂/翻盘状态。
- 如果当前用户已登录，返回 `mySide`、`myBet`。

### 2. PK话题详情

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/pk/topic?topicId=1` 或 `GET /api/pk/topic?slug=pk-hero` |
| 工作量 | M |
| 作用 | 获取单个PK详情页首屏数据 |

要做的事：

- 校验 `topicId/slug` 至少有一个。
- 查询 `PKTopic`、当前 `PKRound`、当前 `PKSeason`。
- 查询当前用户本局下注记录。
- 返回 A/B 奖池、下注人数、赔率、热度、阶段倒计时。
- 返回总战绩、当前连胜、最高连胜。
- 返回最近 N 局结果。
- 返回当前赛季胜场与赛季时间。

### 3. 当前局下注

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/pk/bet` |
| 工作量 | L |
| 作用 | 用户在下注期选择 A/B 阵营下注 |

请求字段：`topicId`、`side=A/B`、`requestId`。

要做的事：

- 校验登录。
- 校验话题存在且启用。
- 查询并锁定当前 `PKRound`。
- 校验当前阶段必须是 `betting`。
- 校验 `side` 合法。
- 校验用户本局未下注；相同 `requestId` 重试要幂等返回已有结果。
- 固定扣除 100 龟币，复用 `UserCoinService.SpendToPool`。
- 创建 `PKBet`。
- 更新 `PKRound.poolA/poolB`、`betCountA/betCountB`。
- 记录下注热度来源 `PKAction(actionType=bet)`。
- 返回下注记录、用户余额快照、最新赔率、最新热度。

### 4. 实时热度

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/pk/heat?topicId=1` |
| 工作量 | M |
| 作用 | 详情页轮询或刷新当前局热度 |

要做的事：

- 查询当前 `PKRound`。
- 返回 `heatA/heatB`、领先方、阶段、倒计时。
- 返回守擂/翻盘状态。
- 可选：触发轻量热度重算，或只读已落库快照。

### 5. PK评论列表

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/pk/comments?topicId=1&side=A&cursor=0` |
| 工作量 | M |
| 作用 | 获取指定阵营的评论战场列表 |

要做的事：

- 查询话题与当前回合。
- 通过 `PKCommentMeta` 按 `topicId/roundId/side` 筛选评论。
- 复用现有 `Comment` 渲染能力。
- 返回评论、点赞数、回复数、热度分、拉踩数、是否已点赞。
- 支持按热度或时间排序，P0 可先按时间。

### 6. 发布PK评论

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/pk/comment/create` |
| 工作量 | L |
| 作用 | 发布带阵营归属的评论 |

请求字段：`topicId`、`side`、`content`、`imageList`、`quoteId`。

要做的事：

- 校验登录、发帖权限、现有 spam 规则。
- 查询当前 `PKRound`。
- 校验当前阶段不是 `cooldown`。
- 校验 `side` 合法；如果用户已下注，建议只能发到已下注阵营。
- 调用或封装 `CommentService.Publish` 创建通用评论。
- 创建 `PKCommentMeta` 绑定 `commentId/topicId/roundId/side`。
- 记录 `PKAction(actionType=comment)`。
- 增量更新或异步重算 `PKRound.heatA/heatB`。
- 返回评论渲染结构与最新热度。

### 7. PK评论回复

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/pk/comment/reply` |
| 工作量 | M |
| 作用 | 回复PK评论，并让回复热度归属父评论阵营 |

请求字段：`commentId`、`content`、`imageList`、`quoteId`。

要做的事：

- 校验父评论存在且有 `PKCommentMeta`。
- 使用父评论的 `topicId/roundId/side` 作为回复归属。
- 校验当前回合仍允许互动。
- 复用 `CommentService.Publish`，创建 `entityType=comment` 的回复。
- 记录 `PKAction(actionType=reply)`。
- 更新父评论回复数与对应阵营热度。

### 8. 拉踩评论

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/pk/downvote` |
| 工作量 | M |
| 作用 | 对对方评论执行拉踩，降低目标评论热度 |

请求字段：`commentId`、`requestId`。

要做的事：

- 校验登录。
- 查询目标评论 `PKCommentMeta`。
- 校验当前阶段允许互动。
- 校验用户本局阵营；只能拉踩对方阵营评论。
- 校验同一用户不能重复拉踩同一评论。
- 创建 `PKAction(actionType=downvote)` 或独立 `PKDownvote`。
- 更新 `PKCommentMeta.downvoteCount`。
- 重算目标评论热度与双方回合热度。
- 返回最新评论热度和回合热度。

### 9. 历史对局

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/pk/history?topicId=1&page=1&pageSize=20` |
| 工作量 | S |
| 作用 | 查看某话题历史每局结果 |

要做的事：

- 分页查询已结算 `PKRound`。
- 返回每局 A/B 热度、胜方、下注人数、评论数、点赞数、结算时间。

### 10. 赛季历史

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/pk/seasons?topicId=1&page=1&pageSize=20` |
| 工作量 | S |
| 作用 | 查看历史赛季记录 |

要做的事：

- 分页查询 `PKSeason`。
- 返回赛季编号、起止时间、A/B 胜场、冠军、总局数。

### 11. 我的PK记录

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/pk/my/bets?page=1&pageSize=20` |
| 工作量 | M |
| 作用 | 用户查看自己参与过的PK下注记录 |

要做的事：

- 校验登录。
- 分页查询当前用户 `PKBet`。
- 关联 `PKTopic`、`PKRound`。
- 返回话题、阵营、下注额、输赢、派奖金额、回合状态。

## 管理端新增接口

### 1. 话题管理列表

| 项 | 内容 |
|----|------|
| 接口 | `GET /api/admin/pk/topic/list` |
| 工作量 | M |
| 作用 | 管理端查看PK话题 |

要做的事：

- 分页查询 `PKTopic`。
- 支持按状态、关键词筛选。
- 附带当前局、当前赛季、总局数、总胜场。

### 2. 新增或编辑话题

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/admin/pk/topic/save` |
| 工作量 | L |
| 作用 | 创建或编辑PK话题 |

要做的事：

- 校验标题、A/B 阵营、slug 唯一。
- 新建话题时同步创建第一赛季和第一回合。
- 编辑时限制不能破坏进行中回合的关键字段。
- 保存排序、状态、展示图等运营字段。

### 3. 启用或停用话题

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/admin/pk/topic/status` |
| 工作量 | M |
| 作用 | 启用或停用PK话题 |

要做的事：

- 校验话题存在。
- 启用时确保存在当前赛季和当前回合。
- 停用后用户端列表隐藏，当前回合不再接受新动作。

### 5. 赛季管理列表

| 接口 | `GET /api/admin/pk/season/list` |
| 工作量 | S |

- 返回胜场、冠军、起止时间、总局数。

### 6. 管理员重算热度

| 项 | 内容 |
|----|------|
| 接口 | `POST /api/admin/pk/recalc_heat` |
| 工作量 | L |
| 作用 | 异常修复或规则调整后重算热度 |

要做的事：

- 校验 `topicId/roundId`。
- 基于 `PKCommentMeta`、`Comment`、`UserLike`、`PKAction` 重算。
- 更新 `PKCommentMeta.heatScore`。
- 更新 `PKRound.heatA/heatB`。
- 如果回合已结算，默认禁止重算；如允许，需要额外审计日志。

默认话题建议用 migration 幂等初始化；如果后续确实需要后台按钮，可再补 `POST /api/admin/pk/init_topics`。

## 现有接口改动

### 1. 评论接口

| 现有接口 | `POST /api/comment/create` |
|----------|---------------------------|
| 建议 | 不直接改动，新增 PK 评论接口封装 |

原因：

- 普通评论不需要阵营和回合。
- 对立PK需要写 `PKCommentMeta`，并更新热度。
- 直接改通用评论接口会让普通评论承担 PK 业务复杂度。

### 2. 回复列表接口

| 现有接口 | `GET /api/comment/replies` |
|----------|----------------------------|
| 建议 | 可复用，必要时新增 PK 专用包装 |

要做的事：

- P0 可直接复用现有回复列表。
- 如果前端需要回复热度、阵营信息，则新增 `/api/pk/comment/replies`。

### 3. 点赞接口

| 现有接口 | `POST /api/like/like`、`POST /api/like/unlike` |
|----------|-----------------------------------------------|
| 建议 | 先复用，后续用事件或 PK 专用接口补热度 |

要做的事：

- 点赞 comment 仍走现有接口。
- 新增点赞事件处理：当点赞对象是 PK 评论时，记录或更新热度。
- 如果需要严格即时热度，可新增 `/api/pk/comment/like` 包装现有点赞服务。

### 4. 金币接口

| 现有能力 | `UserCoinService` |
|----------|-------------------|
| 建议 | 不改用户端金币 API，服务层复用 |

要做的事：

- 新增 `PK_BET_STAKE_IN`、`PK_PAYOUT`、`PK_DRAW_REFUND` 流水类型。
- 在 `PKService` 内通过 `UserCoinService` 完成扣款与派奖。
- 管理后台金币流水无需新增接口，可按 `bizType` 筛选。

## 定时任务相关工作

### PK CronTick

| 项 | 内容 |
|----|------|
| 入口 | `PKService.CronTick()` |
| 工作量 | L |
| 作用 | 自动推进回合与赛季 |

要做的事：

- `betting -> locked`：到达 `lockTime` 后禁止下注。
- `locked -> cooldown`：到达 `endTime` 后计算热度、判胜负、生成结算。
- `cooldown -> betting`：到达 `nextRoundTime` 后创建下一局。
- 赛季结束：归档冠军，创建下一赛季。
- 所有动作必须幂等。

### Scheduler 注册

| 项 | 内容 |
|----|------|
| 文件 | `internal/scheduler/cron.go` |
| 工作量 | S |
| 作用 | 每分钟调用 PK 定时任务 |

要做的事：

- 添加 `PKService.CronTick()` 定时注册。
- 记录开始、结束、错误日志。

## 工作量汇总

| 类型 | 数量 | 说明 |
|------|------|------|
| 用户端新增接口 | 11 | 列表、详情、下注、热度、评论、回复、拉踩、历史、我的记录 |
| 管理端新增接口 | 6 | 话题、回合、赛季、热度重算 |
| 现有接口改动 | 4 | 评论、回复、点赞、金币服务层复用 |
| 定时任务 | 2 | PK CronTick 与 scheduler 注册 |

优先级建议：

- P0：用户端 1、2、3、4、5、6、9、10；管理端 1、2、3、4；定时任务全部。
- P1：用户端 7、8、11；管理端 5、6；点赞事件联动。
- 可选：如不采用 migration 初始化默认话题，再补管理端初始化接口。
