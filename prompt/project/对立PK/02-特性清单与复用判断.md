# 对立PK：特性清单与复用判断

## 特性总表

| 特性 | 说明 | 复用/新开发 | 现有依据或落点 |
|------|------|-------------|----------------|
| PK话题 | 永恒存在，A/B 阵营固定 | 新开发 | 需要 `PKTopic`，不能直接用临时 `Battle` |
| 当前回合 | 72小时一局，含阶段时间 | 新开发 | 需要 `PKRound` 与状态推进 |
| 阶段控制 | 下注期、锁局期、冷却期 | 新开发 | 可复用 scheduler 框架 |
| 单局下注 | 每局每用户一次，100龟币 | 新开发 + 复用金币 | 新建 `PKBet`，扣款复用 `UserCoinService` |
| 动态赔率 | 根据 A/B 投票人数或奖池动态算 | 新开发 | 预测赔率公式不能直接套用 |
| 奖池结算 | 胜方瓜分败方投入 | 新开发 + 复用资金流水 | 可复用资金池/流水模式 |
| 热度计算 | 点赞、评论、拉踩、下注综合 | 新开发 | 现有只有简单 heat/count |
| 评论战场 | A/B 双列评论，评论绑定阵营 | 扩展复用 | 复用 Comment，需新增阵营映射 |
| 回复 | 嵌套回复计入热度 | 复用 | 现有 `entityType=comment` 回复 |
| 点赞 | 评论或PK对象点赞 | 复用 + 扩展统计 | 现有 `UserLike` 支持通用实体 |
| 拉踩 | 降低对方评论热度，同时加己方热度 | 新开发 | 现有没有 downvote 模型 |
| 下注热度 | 本局下注同时为阵营贡献热度 | 新开发 + 复用金币 | 复用 `PKBet` 记录，不新增独立金币动作 |
| 用户权重 | 新/普通/活跃/风险用户权重 | 新开发 | 现有用户与任务数据可辅助 |
| 反作弊系数 | 操作频率、重复内容等 | 扩展复用 | 现有 spam 只覆盖发帖/评论 |
| 赛季 | 30天一季，统计胜场 | 新开发 | 需要 `PKSeason` |
| 历史战绩 | 总胜场、连胜、对局记录 | 新开发 | 由 `PKRound` 派生或冗余 |
| 守擂/翻盘 | 根据上一局胜方与当前领先方提示 | 新开发 | 可由 round + live heat 派生 |
| 话题种子数据 | 16个默认PK话题 | 新开发 | 迁移脚本或运营初始化 |
| 用户侧API | 列表、详情、下注、历史、热度 | 新开发 | 新增 `/api/pk/**` |
| 管理侧API | 话题维护、赛季/回合查看 | 新开发 | 新增 `/api/admin/pk/**` |
| 前端PK页 | 列表、详情、战场、历史 | 新开发 | 现有站点为构建产物，需源码侧配合 |

## 可直接复用的现有能力

### 金币账户与流水

现有 `UserCoinService` 已提供：

- 用户金币账户 `UserCoin`。
- 金币流水 `UserCoinLog`。
- 扣款入池 `SpendToPool`。
- 从池出账 `PayFromPoolToUser`。
- 销毁 `BurnFromPool`。
- 转账 `Transfer`。

对立PK下注与结算都应复用这套账户能力，避免另建余额系统。

### 评论基础能力

现有 `Comment` 支持：

- 通用 `entityType/entityId` 挂载。
- 一级评论与二级回复。
- 图片、引用、IP、UserAgent。
- 评论反垃圾 `spam.CheckComment`。

对立PK可新增 `constants.EntityPKTopic` 或 `pk_topic` 类型，让评论挂到当前 PK 话题或当前回合。

### 点赞基础能力

现有 `UserLike` 支持通用实体点赞：

- 可点赞 topic、article、comment。
- 有唯一约束避免重复点赞。

对立PK可复用 comment 点赞；若需要点赞阵营或回合，也可扩展实体类型。

### 定时任务框架

现有 `internal/scheduler/cron.go` 已注册 battle 每分钟轮巡。

对立PK可以新增 `PKService.CronTick()`，用于：

- 到 48 小时自动从下注期进入锁局期。
- 到 72 小时计算胜负并进入冷却期。
- 冷却 10 分钟后创建下一局。
- 赛季到期后归档并开启新赛季。

## 需要扩展后复用的能力

### 评论阵营绑定

现有 `Comment` 不知道 A/B 阵营。建议不要直接在 `Comment` 表加业务字段，避免污染通用评论。

建议新增 `PKCommentMeta`：

- `commentId`
- `topicId`
- `roundId`
- `side`
- `qualityScore`
- `downvoteCount`
- `heatScore`

### 点赞计入热度

现有 `UserLike` 可以记录点赞，但热度公式需要：

- 用户权重。
- 本局第 k 次操作衰减。
- 反作弊系数。
- 与评论互动热度联动。

建议新增 `PKAction` 或热度快照表，用于记录每次参与热度计算的动作。

### 反垃圾与风控

现有 spam 可用于评论发布前校验，但规则文档中的 AntiSpam 更细。

建议 P0 先使用：

- spam 评论检测。
- 单用户单动作唯一约束。
- 操作频率简单限制。

P1 再加入账号年龄、IP/设备聚类、重复内容检测。

## 必须新开发的能力

### PK领域服务

需要新增 `PKService`，负责：

- 话题查询。
- 当前局状态计算。
- 下注校验与入池。
- 热度重算或增量计算。
- 回合结算。
- 赛季结算。
- 历史统计维护。

### PK数据模型

至少需要：

- `PKTopic`
- `PKRound`
- `PKBet`
- `PKRoundSettlement`
- `PKSeason`
- `PKCommentMeta`
- `PKAction` 或 `PKHeatEvent`

### PK接口

新增 `/api/pk/**`：

- 话题列表。
- 话题详情。
- 当前局下注。
- 实时热度。
- 对局历史。
- 赛季历史。

### 拉踩与下注热度

现有系统没有拉踩；下注热度需要在 `PKBet` 基础上纳入热度计算。

建议新建：

- `PKDownvote`

也可以统一为 `PKAction`，用 `actionType=downvote/bet` 区分热度来源。
