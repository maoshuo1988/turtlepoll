# 对立PK：实施路线

## P0 实施顺序

### 1. 后端模型与迁移

新增 PK 相关模型：

- `PKTopic`
- `PKRound`
- `PKSeason`
- `PKBet`
- `PKSettlementItem`
- `PKCommentMeta`

同时新增默认种子话题迁移。

### 2. 基础服务

新增 `PKService`：

- 获取话题列表。
- 获取话题详情。
- 初始化当前赛季与当前局。
- 下注。
- 当前热度查询。
- 回合结算。
- 创建下一局。

### 3. 金币扣款与结算

下注复用 `UserCoinService.SpendToPool`。

结算复用：

- `UserCoinService.PayFromPoolToUser`
- `UserCoinLog`

注意事项：

- 单用户单局只能下注一次。
- 下注接口必须带 `requestId` 并幂等。
- 结算必须以 `roundId` 幂等。

### 4. 评论战场

复用 `CommentService.Publish` 或封装 `PKCommentService.Publish`。

发布后写入 `PKCommentMeta`：

- 绑定 topic。
- 绑定当前 round。
- 绑定 side。

回复沿用现有 `entityType=comment`，但热度计算时需要回溯父评论对应的 `PKCommentMeta`。

### 5. 热度计算

P0 建议采用可解释简化版：

```text
commentHeat = min(3 + log(1 + likeCount + 1.5 * replyCount), 20)
sideHeat = sum(commentHeat) + likeCount
```

先保证可重算、可落库、可测试。

P1 再加入用户权重、频率衰减、时间权重、质量分、AntiSpam、拉踩、下注热度系数。

### 6. 定时任务

在 `internal/scheduler/cron.go` 注册 `PKService.CronTick()`。

处理：

- `betting -> locked`
- `locked -> cooldown`
- `cooldown -> next betting round`
- 赛季结束归档

### 7. API 与文档

新增：

- `internal/controllers/api/pk_controller.go`
- `internal/controllers/admin/pk_controller.go`
- `docs/api/pk.md`

## 风险与未决问题

### 平局规则

原始规则未定义平局。建议 P0 使用平局退款。

### 评论挂 topic 还是 round

建议评论挂 `pk_topic`，通过 `PKCommentMeta.roundId` 绑定当前局。

理由：

- 话题长期存在，用户看历史评论更自然。
- 当前局和历史局可通过 meta 筛选。
- 回复仍可复用通用评论。

### 结算是否自动入账

建议 P0 自动入账，减少用户操作。

开战广场采用“结算单 + 提取”，但对立PK是高频小额玩法，手动提取会增加摩擦。

如果产品希望统一资金池审计，也可保留结算明细并自动执行 `PayFromPoolToUser`。

### 下注是否允许追加

当前统一话术后，不再保留独立金币动作；下注既是资金动作，也是阵营热度来源。

建议：

- P0：每用户每局只允许下注一次，固定 100 龟币。
- P0：下注金额进入奖池，并按下注金额计算下注热度。
- P1：如需追加下注，必须重新定义单用户单局限制、赔率展示和结算口径。

### 风控分期

完整 AntiSpam 不适合 P0 一次完成。

建议：

- P0：唯一约束、频率限制、现有 spam。
- P1：用户权重、账号年龄、重复内容。
- P2：IP/设备聚类、异常热度审计。
